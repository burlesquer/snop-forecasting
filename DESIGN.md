## Accessibility Specs (added 2026-05-27)

- **키보드 네비게이션**: 모든 인터랙티브 요소 `Tab` 도달 가능. 토글 = `Space`, 탭 = `←→`, 드롭다운 = `↑↓+Enter`. focus ring = 2px accent outline + 2px offset (border와 구분).
- **ARIA landmarks**: `<header>` + `<nav aria-label="탭">` + `<main>` + `<footer>`. 각 위젯은 `<section aria-labelledby="...">`로 묶임.
- **차트 a11y**: Recharts `aria-label` 직접 명시. SVG 차트에 텍스트 폴백 `<title>` + `<desc>`. 시계열 차트는 데이터 테이블 토글 버튼 제공 (스크린리더용).
- **터치 타겟**: 모든 버튼/토글 최소 44×44px. KPI 카드 클릭 영역 = 카드 전체.
- **색대비**: 본문 텍스트(text:bg) ≥ 7:1 (WCAG AAA). 큰 텍스트 ≥ 4.5:1. accent rose는 본문에 안 쓰고 시각 강조만.
- **prefers-reduced-motion**: 차트 entrance, 토글 transition 모두 `@media (prefers-reduced-motion: reduce)`에서 즉시 적용 (transition: none).
- **국문 폰트**: Pretendard는 한글 hinting 우수, 영문/숫자 비례. 별도 영문 폰트 안 섞음.



- **ML 라이브러리**: `nixtla mlforecast` 채택. Naive/MA/LightGBM/quantile을 동일 API로 처리. CV 루프 추상화. 직접 LightGBM 핸드롤 폐기.
- **반응형**: 완전 반응형. 모든 breakpoint (320/768/1024/1440)에서 동작. 시뮬레이터는 모바일에서 simplified UI.
- **브랜딩**: 전면 브랜드 시스템 구축. 토큰화된 컬러/타이포/스페이싱 + 아이콘 세트 + wordmark + 일러스트레이션. **`/plan-design-review` 실행을 다음 단계로 강력 권장.**
- **에러 처리**: 모든 ML 단계에 try/except + 명시적 로그. 프런트엔드는 `<EmptyState />` 단일 컴포넌트로 일관 처리. (Section 2 Error Map 참조)
- **테스트 최소선**: Python pytest 5~8개 (quantile order, schema, simulator math) + Playwright 2~3개 (UI smoke).
- **타입 정의**: `lib/types.ts` + Python `pydantic` 모델로 JSON 스키마 양쪽 명시.

## Problem Statement

S&OP의 핵심 개념(SKU별 수요예측, 안전재고, 서비스레벨, 재고회전율)을 손으로 만져볼 수 있는 데모를 만든다.

면접/포트폴리오 자리에서 "AI 기반 S&OP 시스템 구축 경험"을 구체적으로 보여줄 수 있는 형태. Kaggle M5 Forecasting-Accuracy 데이터셋을 화장품 시나리오로 리프레이밍해서, **경영자 KPI 대시보드**와 **데이터과학자 분석 뷰**를 동시에 가진 웹앱으로 만든다.

## What Makes This Cool

- 단순 예측 모델 데모가 아니라, **S&OP 의사결정 워크플로 전체**를 시뮬레이션
- **이중 관점 대시보드**: 같은 데이터를 경영자 시선(KPI, 위험 SKU)과 분석가 시선(MAPE/WAPE, 잔차, feature importance)으로 동시에 제시
- **실제 비즈니스 영향 정량화**: 결품 손실, 과잉재고로 묶인 현금, 적중률 1%p 개선의 가치를 원화로 환산
- 공유 가능한 URL로 면접 자리에서 노트북 열어 즉시 시연 가능
- "AI 아이디어는 있는데 구현을 못한다"고 한 회사들에게 정확히 원하는 형태의 결과물

## Constraints

- Windows 11 환경 (개발자 PC)
- M5 데이터는 Kaggle에서 별도 다운로드 (약 450MB, 라이선스상 repo에 포함 불가)
- Vercel free tier 내에서 호스팅
- 한국어 UI (경영자용 카피)
- 사전 계산 모델 (live ML inference 불필요 - 배포 단순화)

## Premises

1. **1개 매장 × 약 50개 SKU 서브셋 사용** — 풀 30K 시계열은 데모용으로 과함. 카테고리 다양성 확보를 위해 4개 카테고리에서 분산 추출.
2. **화장품 리프레이밍은 표시 레이어만** — M5는 Walmart 식료품/잡화. 데이터 자체는 그대로, UI에서 카테고리/SKU명만 화장품 시나리오로 매핑.
3. **예측 horizon 28일** — M5 대회 표준이자 S&OP 월간 사이클 매칭.
4. **모델 = LightGBM (글로벌) + Naive 베이스라인 + Moving Average** — M5 우승자 방식. 베이스라인 대비 ML 성능 차이가 S&OP의 "왜 AI인가" 핵심 메시지.
5. **Pre-computed JSON 데이터 구조** — Python 추론은 빌드 타임에만, 프런트엔드는 정적 JSON을 읽음. 배포가 가벼움.

## Approaches Considered

### Approach A: Streamlit 모놀리스 (Completeness 6/10)
- 단일 app.py에 데이터 로딩 → 예측 → 재고계산 → UI까지
- **Pros**: 30분 안에 동작. Python만으로 끝.
- **Cons**: 시스템 구조가 안 보임. 웹 URL 공유에 약함 (Streamlit Cloud는 가능하나 인상 약함). 면접 임팩트 낮음.

### Approach B: 모듈화 파이프라인 + Streamlit (Completeness 9/10)
- src/data, src/models, src/inventory, src/dashboard 분리
- **Pros**: 학습 깊이 높음. 엔지니어링 구조 보임.
- **Cons**: 여전히 Streamlit. 웹 공유시 데모 임팩트 부족.

### Approach C: Next.js 대시보드 + Python 사전 계산 ★선택 (Completeness 10/10)
- Next.js 16 App Router + shadcn/ui + Tailwind, Vercel 호스팅
- Python 파이프라인이 빌드 타임에 모든 forecast/KPI를 JSON으로 출력
- **Pros**: 면접 임팩트 최대. URL 한 줄로 공유. 모던 스택. 풀스택 능력 시연.
- **Cons**: 프런트엔드 코드량 증가. Windows에 Node 24+ 셋업 필요.

## Recommended Approach: C

## What I noticed about how you think

- "샘플로 만들고 싶어"라고 하셨지만 옵션 셋에서 풀 시뮬레이션을 골랐어요. 머릿속의 "샘플" 정의가 이미 "보여줄 수 있는 완성품"이었던 것 같아요. 이 자기 인식의 불일치를 본인이 알아채는 게 중요합니다 — 다음번엔 처음부터 "프로토타입" 또는 "MVP"라고 부르세요.
- 관객 질문에서 "둘 다 살리고 싶다"고 하신 게 인상적. 보통 한쪽을 포기하는데, 두 페르소나를 동시에 설계할 수 있다는 건 실무 PM 사고에 가까워요. 이건 데이터 사이언티스트한테 흔치 않은 능력.
- 구현 옵션에서 "클래시" 옵션(C)을 선택. 학습 효율보다 "이걸로 뭔가 보여주겠다"는 의도가 강함. 면접 무기로서의 가치를 정확히 인식하고 있다는 신호.
- 직전 S&OP 개념 대화에서, 추상 개념(서비스레벨, 안전재고)을 SKU 단위 + 화장품 업계 시나리오로 자연스럽게 변환하셨어요. 도메인 ↔ 데이터 사이의 번역 능력은 데이터 사이언티스트의 핵심 자질입니다.
